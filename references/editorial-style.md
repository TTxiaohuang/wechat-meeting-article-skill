# 编辑规范

撰写或修改文章内容前，请先阅读本文件。

## 编辑优先级

1. 保证事实准确性与发言人归属。
2. 让文章读起来像一篇打磨过的公众号推文，而不是素材提取报告。
3. 源文件名和本地路径不要出现在文章正文中。
4. 灵活使用环节：不强制套用模板，缺失素材的环节直接省略。非标会议使用 `sessions` 数组自定义顺序。
5. `source` 字段仅用于溯源，不在文章中展示。仅在用户明确要求时才创建 `source_trace.md`。
6. 脚手架文件和 `article_notes` 仅作素材清单参考，最终文章必须通过阅读提取后的原始素材来撰写。
7. `check_article_json.py` 报错即阻断，修复后才能继续。不要静默跳过任何报错。

## 非标会议

当会议不遵循标准流程（英语交流 → 文献分享 → 时政交流 → 自由讨论）时，**用 `sessions` 数组替代 `sections`**。渲染器按数组顺序输出；预定义类型有专用样式，其他 type 走通用渲染。

**判断标准**：如果标准顺序套不上这次会议的实际内容，就用 `sessions`。

**自定义 session 结构**：
```json
{
  "type": "graduation_sharing",
  "title": "毕业论文分享",
  "topic": "论文进展与心得",
  "intro": "本次环节邀请三位同学分享毕业论文进展。",
  "images": [],
  "items": [
    {"speaker": "李四", "text": "分享内容摘要。", "source": "transcript.docx"}
  ],
  "closing": ""
}
```

**编辑要求**：
- 自定义环节的 `items` 同样需要提炼概括，不能照搬录音原文
- `intro` 简要说明该环节的背景和参与人
- `closing` 可选，用于该环节的小结
- `type` 值用英文下划线命名（如 `graduation_sharing`、`career_sharing`、`thesis_defense`）
- 如果某个自定义环节素材很少，考虑合并到相邻环节或省略

## 英语交流

- 如果提供了英语发言稿，使用英文原文作为卡片正文。
- 不要翻译、润色或总结英语发言，除非用户明确要求。
- 每人一张横向滚动卡片。
- 卡片前可以加一小段中文引言。
- 文件名仅保留在 `article.json` 的 `source` 字段中，不出现在文章正文。

## 文献分享

为每篇论文撰写有实质内容但简洁的学术介绍，优先使用以下结构：

- `background`：论文的研究背景和重要性
- `research_question`：核心问题或假设
- `methods_data`：数据、模型、识别策略、样本或分析方法
- `findings`：2-4 条核心发现
- `discussion_value`：对本次组会或课题组研究的启发
- `comments`：组会上老师和同学的观点评论

避免空泛表述如"本文具有重要的参考价值"，除非后面跟具体说明。不要在论文设计不支持因果推断时过度宣称因果结论。

## 完整性检查

渲染前，将最终文章与提取素材逐项对比：

- `_meta.intake_gate` 缺失或不完整 → 渲染前停下，补全缺失决策
- 英语发言稿有很多人但 `article.json` 只有一两个 → 回查提取，不要发布
- 转录稿很长但自由讨论只有一句 → 回查转录稿
- 论文提取有几千字但文献分享只有几行 → 扩充文献笔记
- PDF 提取页数少于总页数 → 不要假设全文已读，补充提取或注明限制
- PDF 提取文本页为 0 → 视为扫描件，无法直接使用
- `article.json` 含 `_meta.scaffold_generated` → 必须重写为最终版本再渲染
- 提取素材中有明确的喜讯、通知、里程碑、论文录用、证书等 → 纳入 `custom_sections`，除非有歧义或隐私问题

## 转录稿处理

录音转录文件常有识别错误，不要把 ASR 噪声当成确定事实。

- 优先使用归纳式讨论摘要，而非大段直接引用转录稿。
- 将参会者姓名、论文标题、奖项名称、方法、日期、机构等与非 ASR 材料交叉校验。
- 发言人身份不确定时，用"老师"、"同学讨论"、"会议讨论"等模糊归属。
- 精确的姓名和奖项细节仅在有多个来源支撑或明确写在准备文件中时才使用。
- 当 ASR 噪声影响公开事实或隐私敏感细节时，向用户确认。

## 自由讨论 vs 文献讨论

**这两个环节的内容不能重叠。** 文献讨论评论（论文方法批评、方法论质疑、主讲人回应等）留在 `sections.literature_sharing.papers[].comments` 中。自由讨论涵盖论文之外的更广泛话题。

**自由讨论必须编辑提炼：**

- 归纳和浓缩口语内容，不要粘贴原始转录稿。
- 去掉口头禅、重复、跑题和非正式表达。
- 保留发言人的核心观点和可操作的洞见，而非原话。
- 老师的长篇评论应提炼为 2-4 句简洁的核心信息。
- 如果自由讨论中引用了文献分享已讨论的论文，只保留能带来新视角的部分（如可推广的经验、对未来研究的建议），不要复述文献讨论。
- **同一发言人的多段发言必须合并为一个 item**。如果老师在讨论中说了多段话，将它们提炼合并为一张卡片，不要拆成多张。

**自由讨论必须包含简短的 `intro`**（1-2 句话），概括本环节讨论的话题，让读者快速了解。

## 可选插页

插页要克制使用。文章本质仍是读书分享会推文，插页应该是会议流程中的短暂亮点。

- `honor_news`：学生获奖、竞赛成绩、项目立项、论文录用等喜讯。优先放在导读之后或文献分享之前。
- `announcement`：会议通知、活动预告、投稿提醒等。优先放在自由讨论之前或会议小结之前。
- `milestone`：课题进展、发表里程碑、数据发布等。放在相关环节附近。
- `note`：不值得单独成节的简短附注。

当文件名或提取文本明确指示喜讯类内容（如"喜讯"、"祝贺"、"获奖"、"立项"、"入选"、"录用"、"证书"、"通知"），自动纳入插页。当姓名、奖项等级、证书图片、隐私或公开发布权限存疑时，向用户确认一次。

## 图片使用

仅在图片有助于读者理解会议内容时使用：

- 会议照片
- PPT 标题页或关键幻灯片截图
- 论文框架图/机制图/核心结果图（需素材中提供）

不要编造数据图表、回归表格或论文图片。如果没有合适的图片，在最终回复中给出图片建议而非伪造。

当素材文件夹中有用途不明确的图片时，列出文件名并询问用户：
1. 这些图片是什么？（会议照片、截图、证书等）
2. 放在文章什么位置？

不要根据文件名猜测图片内容。不要因为用途不明确就静默跳过。

## 公众号署名

公众号名称：郑而研资。文章末尾自动添加"本期记录由郑而研资整理"及主持/编辑署名，无需询问用户。`"theme": "zhengeryanzi"` 是默认值，直接写入 `article.json` 即可。渲染器会自动添加 SVG 章节标记和结尾署名。

## 模板与配色

模板选项：

- `classic`：默认简洁风格
- `notebook`：读书笔记风格，柔和色带+左侧色条
- `campus`：明亮校园风格
- `magazine`：编辑排版风格，开头节奏更强
- `briefing`：报告风格，分隔线和编号标签更突出

配色选项：

- `classic`：柔和蓝+暖色注记
- `forest`：学术绿+柔和米色
- `blueprint`：蓝色报告色调
- `warm`：克制的棕绿色读书笔记色调
- `ink`：灰黑色学术风格
- `sunrise`：暖珊瑚+绿色，适合清新校园推文
- `mono`：接近单色的极简色调
- `sakura`：樱花粉+暖白
- `ocean`：深海蓝+浅青色

旧名称（`journal`、`minimal`、`warm-note`、`fieldnote`）通过别名映射到最近的模板。

SVG 用于渲染器内置的装饰元素（章节标记、引用标记、头像），用户无需操作。自定义 SVG 动画仅在明确要求并测试后使用。

结尾署名优先使用 `meta.host` 和 `meta.editor`，渲染为"主持：..."和"推文编辑：..."。有署名信息时不要用通用口号。

图片放置建议：

- 英语交流：可选的 topic 卡片
- 文献分享：论文框架图、PPT 关键幻灯片、方法流程图
- 讨论总结：会议照片或阅读清单图片

## 微信正文风格

- 标题下方放简短的导读摘要。
- 紧凑的章节标题和清晰的段落节奏。
- 讨论评论用引用块。不要把每个文献子字段都放进彩色卡片中，普通文本加小标题通常更易读。
- 装饰克制：细分割线、小 SVG 标记、浅色引用背景、偶尔的图片分隔。避免秀米式密集边框、贴纸式装饰、闪烁效果和装饰堆砌。
- 不要在正文中出现 `source`、本地文件名、JSON key、提取备注或脚本名等技术标签。
- 用词温暖且学术：足够通俗适合公众阅读，但不要营销化。
- 最终回复中包含建议的公众号标题和摘要，因为微信编辑器的标题和摘要字段需要手动填写。
